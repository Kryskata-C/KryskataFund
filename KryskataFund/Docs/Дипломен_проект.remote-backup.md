# Увод

Дипломният проект представя разработката на KryskataFund \-- уеб базирана платформа за колективно финансиране (crowdfunding), реализирана с технологията ASP.NET Core 8 и архитектурния модел MVC (Model-View-Controller). Платформата дава възможност на потребителите да създават благотворителни кампании за набиране на средства в различни категории \-- образование, здраве, животни, творчески проекти, технологии, общност и други \-- както и да правят дарения чрез интеграция с платежната система Stripe.

Основната цел на проекта е да демонстрира приложението на съвременни технологии и практики за изграждане на сигурно, тествано и поддържаемо уеб приложение. За постигането на тази цел са формулирани следните задачи:

1\. Проектиране и реализация на MVC архитектура с ясно разделение на отговорностите \-- модели, контролери, изгледи и сервизен слой с интерфейсна абстракция.

2\. Внедряване на многослойна защита срещу основните уеб уязвимости (OWASP Top 10), включваща BCrypt хеширане на пароли, CSRF защита, XSS санитизация и параметризирани заявки.

3\. Изграждане на обширен набор от автоматизирани тестове \-- модулни, интеграционни и специализирани тестове за сигурност.

4\. Конфигуриране на CI/CD (Continuous Integration / Continuous Delivery) процес чрез GitHub Actions за автоматично компилиране, тестване и генериране на Docker образ.

5\. Интеграция с външни услуги \-- Stripe за обработка на плащания и Resend за изпращане на транзакционни имейли.

Платформата KryskataFund включва функционалност за регистрация и автентикация на потребители, създаване и управление на кампании с етапни цели и обновления, обработка на дарения (еднократни и рекурентни), лични съобщения между потребители, търсене с автоматично довършване, класация на кампании и дарители, административен панел с аналитика и геймификационни елементи.

Технологичният стек включва ASP.NET Core 8, Entity Framework Core 8 с PostgreSQL за продукция и SQLite за разработка, Razor за генериране на изгледите, Bootstrap 5 с персонализирана тъмна тема за фронтенда, BCrypt.Net-Next и HtmlSanitizer за сигурност, xUnit, FluentAssertions и Moq за тестване, и GitHub Actions с Docker за CI/CD.

Разработеното приложение демонстрира способността за самостоятелно проектиране, имплементация и осигуряване на качеството на цялостен уеб продукт, следващ утвърдени практики в софтуерната индустрия.

## Глава 1 Проучвателна част

В настоящата глава е представен анализ на съвременните платформи за колективно финансиране (crowdfunding), използваните технологии за уеб разработка и обосновката за избора на технологичен стек за проекта KryskataFund.

### 1.1. Колективно финансиране \-- общ преглед

Колективното финансиране (crowdfunding) е метод за набиране на средства, при който голям брой хора допринасят с малки суми за финансирането на проект, кауза или инициатива. През последните години тази форма на финансиране се превърна в значима алтернатива на традиционните методи за набиране на капитал. Глобалният пазар на crowdfunding платформи продължава да расте, като водещи играчи са GoFundMe, Kickstarter и Indiegogo.

Съществуват няколко основни модела на колективно финансиране: дарителски (donation-based), при който средствата се даряват безвъзмездно; наградно-базиран (reward-based), при който дарителите получават продукт или услуга; инвестиционен (equity-based), при който дарителите получават дял от компанията. Проектът KryskataFund реализира дарителския модел, при който потребителите финансират благотворителни кампании без очакване за материална възвращаемост.

### 1.2. Анализ на съществуващите решения

За формулирането на изискванията към KryskataFund бяха проучени следните популярни платформи:

GoFundMe е най-голямата глобална платформа за набиране на средства за лични каузи и благотворителност. Основните й предимства включват лесен процес на създаване на кампании, социална интеграция за споделяне и нулева комисионна за платформата (потребителите плащат само транзакционни такси). Недостатъците включват ограничена аналитика за създателите и липса на етапни цели (milestones).

Kickstarter е платформа за финансиране на креативни проекти с наградно-базиран модел. Отличава се с модел \"всичко или нищо\" (all-or-nothing), при който средствата се събират само при постигане на целевата сума. KryskataFund черпи вдъхновение от концепцията за етапни цели и обновления на кампаниите.

Indiegogo предлага гъвкав модел на финансиране, при който създателите могат да запазят набраните средства независимо дали целта е постигната. Платформата поддържа разширена аналитика и интеграция с маркетингови инструменти.

На база анализа бяха идентифицирани ключови функции, които KryskataFund трябва да предостави: лесно създаване на кампании с категории и визуализация на прогреса, сигурна обработка на плащания, етапни цели и обновления, социални функции (следване, коментари, съобщения) и аналитика за създатели.

### 1.3. Избор на технологичен стек

За реализацията на проекта беше избран следният технологичен стек:

ASP.NET Core 8 е модерна, високопроизводителна рамка за изграждане на уеб приложения от Microsoft. Тя е с отворен код, крос-платформена и предоставя вградена поддръжка за MVC архитектура, Dependency Injection, сесии, антифалшификационни токени и Entity Framework Core. Версия 8 е Long-Term Support (LTS) версия с гарантирана поддръжка до ноември 2026 г.

Entity Framework Core 8 е ORM (Object-Relational Mapper), който позволява работа с базата данни чрез C# обекти вместо директен SQL. Поддържа Code First подход за миграции, LINQ заявки с автоматична параметризация (защита от SQL инжекции) и множество доставчици за различни СУБД (PostgreSQL, SQLite, SQL Server).

PostgreSQL е мощна система за управление на релационни бази данни с отворен код. Избрана е за продукционната среда поради високата й надеждност, разширена поддръжка на JSON типове данни, пълна ACID съвместимост и безплатна лицензия. За разработка се използва SQLite, който не изисква отделен сървър.

Stripe е водеща платежна платформа, предоставяща API за обработка на онлайн плащания. Избрана е поради отличната документация, .NET SDK (Stripe.net v50.4.1), поддръжка на Checkout Sessions за сигурни плащания и рекурентни дарения.

Bootstrap 5 е CSS рамка за изграждане на респонсивни потребителски интерфейси. Приложението използва Bootstrap 5 заедно с персонализирана тъмна тема, дефинирана в kryskatafund.css с CSS custom properties за последователно визуално оформление.

\- Качването на изображения се ограничава до URL адрес или единичен файл.

## Глава 3 [Проектен цикъл на приложението](#_heading=h.n0tmrxdwmtq5)

В настоящата глава е описан проектният цикъл на приложението KryskataFund, включващ архитектурния дизайн, организацията на кода, базата данни, сигурността, тестването и CI/CD процесите.

### 3.1. Архитектура на приложението

Приложението KryskataFund е разработено по архитектурния модел MVC (Model-View-Controller), реализиран чрез рамката ASP.NET Core 8. Този модел осигурява ясно разделение на отговорностите:

Моделите (Models) представляват домейнните обекти на приложението и включват 14 класа: User, Fund, Donation, FundUpdate, FundMilestone, FundComment, Message, UserFollow, FundCollaborator, DeadlineExtension, RecurringDonation, както и ViewModels \-- SignUpViewModel, CreateFundViewModel и HomeViewModel. Всеки модел дефинира структурата на данните и правилата за валидация чрез Data Annotations.

Контролерите (Controllers) обработват HTTP заявките и координират взаимодействието между моделите и изгледите. Приложението включва 13 контролера: AccountController (автентикация), FundsController (управление на кампании), DonationsController (обработка на дарения), AdminController (администрация), HomeController (начална страница), SearchController (търсене), MessagesController (съобщения), FundUpdatesController, FundMilestonesController, FundCollaboratorsController, FundCommentsController, FundAnalyticsController и ErrorController.

Изгледите (Views) са реализирани чрез Razor шаблони (.cshtml), които комбинират C# код с HTML за динамично генериране на съдържание. Те са организирани по контролери в директории: Account, Funds, Admin, Home, Search, Messages, Error и Shared.

Сервизният слой (Services) капсулира бизнес логиката и предоставя абстракция чрез интерфейси. Реализирани са четири основни сервиса: UserService (управление на потребители), FundService (управление на кампании), DonationService (управление на дарения) и EmailService (изпращане на имейли). Всеки сервис има съответен интерфейс (IUserService, IFundService и т.н.), което позволява лесна подмяна на имплементацията и улеснява тестването чрез mock обекти.

Допълнителни слоеве включват филтрите за оторизация (Filters), константите (Constants) и статичните помощни класове (PasswordHasher).

Директорийната структура на проекта е следната:

### 3.2. База данни и Entity Framework Core

Приложението използва Entity Framework Core 8 като ORM (Object-Relational Mapping) за работа с базата данни. Контекстът ApplicationDbContext дефинира 11 DbSet колекции, всяка от които съответства на таблица в базата данни:

\- Users \-- потребителски акаунти с хеширани пароли и администраторски флаг

\- Funds \-- кампании за набиране на средства с всички свързани атрибути

\- Donations \-- записи за индивидуални дарения, свързани с фонд и потребител

\- FundUpdates \-- обновления, публикувани от създатели на кампании

\- FundMilestones \-- етапни цели с целева сума и статус на достигане

\- FundComments \-- коментари на потребители към кампании

\- Messages \-- лични съобщения между потребители с поддръжка на споделяне на фондове

\- UserFollows \-- следвания на кампании от потребители

\- FundCollaborators \-- колаборатори с роли за управление на кампании

\- DeadlineExtensions \-- заявки за удължаване на крайния срок на кампания

\- RecurringDonations \-- рекурентни (периодични) дарения с възможност за отмяна

Схемата на базата данни е създадена чрез Code First подход с 10 миграции, покриващи последователното добавяне на таблици и връзки. Уникален индекс е наложен върху полето Email в таблицата Users за предотвратяване на дублирани регистрации.

Средата за разработка използва SQLite база данни (kryskatafund.db), докато продукционната среда е конфигурирана за PostgreSQL чрез Npgsql доставчик. Приложението автоматично парсва DATABASE_URL променливата на средата за съвместимост с облачни платформи като Heroku.

За начално зареждане на данни е реализиран DbSeeder клас, който създава 11 тестови потребителя (включително администратор), 10 примерни кампании в 8 категории, 69 дарения и допълнителни обновления и етапни цели. Всички пароли в seed данните се хешират с BCrypt чрез споделения PasswordHasher клас.

### 3.3. Сигурност на приложението

Сигурността е приоритет при разработката на KryskataFund. Внедрени са множество защитни механизми, покриващи основните рискове от OWASP Top 10:

### 3.3.1. Хеширане на пароли с BCrypt

Всички потребителски пароли се хешират с алгоритъма BCrypt преди записване в базата данни. BCrypt е специално проектиран за хеширане на пароли \-- той е адаптивно бавен (чрез конфигурируем cost factor), което прави brute-force атаките непрактични. Всеки хеш включва уникална сол (salt), генерирана автоматично от библиотеката BCrypt.Net-Next.

Централизираният клас PasswordHasher (в Services/PasswordHasher.cs) осигурява единна точка за хеширане и верификация. Класът поддържа и наследен SHA256 формат \-- при вписване на потребител със SHA256 хеш, паролата автоматично се мигрира към BCrypt. Детекцията на формата се извършва чрез проверка на префикса \"\$2\" (характерен за BCrypt хешове), като всичко останало се третира като Base64-кодиран SHA256 хеш.

Автоматичната миграция е реализирана в AccountController \-- при успешна верификация с наследен формат, паролата се прехешира с BCrypt и се записва обратно в базата данни, без потребителят да забележи промяната.

### 3.3.2. Защита срещу Cross-Site Request Forgery (CSRF)

CSRF защитата е реализирана на глобално ниво чрез AutoValidateAntiforgeryTokenAttribute, регистриран в Program.cs. Това означава, че всички POST, PUT и DELETE заявки автоматично изискват валиден Anti-Forgery Token, без необходимост от изрично анотиране на всяко действие.

За формуляри, използващи стандартни HTML форми, Razor автоматично генерира скрито поле с токена. За AJAX заявки (fetch API), токенът се извлича от мета таг в страницата и се добавя като RequestVerificationToken хедър. Тази защита е внедрена във всички интерактивни елементи: формуляри за дарение, администраторски действия, изпращане на съобщения, следване на кампании и персонализация на Buddy.

### 3.3.3. Защита срещу Cross-Site Scripting (XSS)

За предотвратяване на XSS атаки е внедрена библиотеката HtmlSanitizer (v9.0.892). Описанията на кампаниите, които позволяват HTML форматиране, преминават през санитизиращ филтър при създаване и редактиране:

Description = new HtmlSanitizer().Sanitize(model.Description);

HtmlSanitizer премахва потенциално опасни HTML елементи (script, iframe и др.), атрибути за обработка на събития (onerror, onload и др.), JavaScript URI схеми и CSS expressions. Библиотеката запазва безопасно HTML форматиране (параграфи, списъци, заглавия), позволявайки на създателите на кампании да оформят текста, без да компрометират сигурността.

### 3.3.4. Защита срещу SQL инжекции

Всички заявки към базата данни преминават през Entity Framework Core, който автоматично параметризира потребителския вход. В проекта не се използва суров SQL код \-- всички заявки са изразени чрез LINQ (Language Integrated Query). Например, SearchController използва .AsEnumerable().Where() за клиентско филтриране след безопасно извличане на данни от базата.

### 3.3.5. Персонализирани филтри за оторизация

Вместо стандартния \[Authorize\] атрибут на ASP.NET Core, проектът използва два персонализирани Action Filter атрибута:

RequireSignInAttribute (Filters/RequireSignInAttribute.cs) \-- проверява дали потребителят е вписан чрез сесийния ключ IsSignedIn. При невписан потребител, заявката се пренасочва към страницата за вписване. Атрибутът е приложен върху действията на AccountController (Profile, MyFunds, MyDonations, Following), MessagesController и действието Create на FundsController.

RequireAdminAttribute (Filters/RequireAdminAttribute.cs) \-- проверява дали потребителят има администраторски права чрез сесийния ключ IsAdmin. При неупълномощен достъп, потребителят се пренасочва към страницата Forbidden (HTTP 403), а не към началната страница, което е по-информативно и следва добрите практики за HTTP семантика.

### 3.3.6. Конфигурация на сесиите

Сесиите на потребителите са конфигурирани с множество защитни мерки:

\- 30-минутен idle таймаут за автоматично прекратяване на неактивни сесии

\- HttpOnly бисквитки, които предотвратяват достъпа до сесийните данни от JavaScript

\- SameSite=Strict политика, която предотвратява изпращането на бисквитки при cross-site заявки

\- SecurePolicy, следваща протокола на заявката (HTTPS в продукция)

\- Distributed Memory Cache за сървърно съхранение на сесийните данни

Сесийните ключове са централизирани в Constants/SessionKeys.cs за предотвратяване на типографски грешки и улесняване на поддръжката.

### 3.5. CI/CD процес (Непрекъсната интеграция и доставка)

За автоматизиране на процеса по компилация, тестване и разгръщане е конфигуриран CI/CD пайплайн чрез GitHub Actions. Конфигурацията се намира във файла .github/workflows/ci.yml и включва две задачи (jobs):

**Задача build-and-test:**

Тази задача се изпълнява при всяко качване на код (push) или заявка за сливане (pull request) към клоновете master и main. Стъпките включват:

1\. Checkout на изходния код чрез actions/checkout@v4

2\. Инсталиране на .NET 8.0.x SDK чрез actions/setup-dotnet@v4

3\. Възстановяване на NuGet зависимостите (dotnet restore)

4\. Компилация в Release конфигурация (dotnet build \--no-restore -c Release)

5\. Изпълнение на всички тестове (dotnet test \--no-build -c Release \--verbosity normal)

**Задача docker:**

Тази задача се изпълнява само при успешно преминаване на build-and-test и само при push към master клона. Тя генерира Docker образ на приложението чрез docker build -t kryskatafund, готов за разгръщане в контейнерна среда.

Пайплайнът осигурява автоматичен контрол на качеството \-- всеки комит, който нарушава компилацията или провалва тест, е незабавно видим за екипа. Това следва принципите на Continuous Integration, описани от Martin Fowler.

За локална разработка е осигурена и Docker Compose конфигурация (docker-compose.yml), която стартира SQL Server 2022 контейнер с предварително конфигурирани портове и обеми за данни.

### 3.6. Подобрения в качеството на кода

В хода на разработката бяха извършени редица подобрения за повишаване на поддържаемостта и четимостта на кода:

Екстракция на константи: Сесийните ключове (SessionKeys.cs) и цветовете на категориите (CategoryColors.cs) бяха извлечени в отделни константни класове в директория Constants. Това елиминира \"магическите стрингове\" и намалява риска от типографски грешки.

Въвеждане на ViewModels: Началната страница (HomeController) беше рефакторирана от ViewBag към типизиран HomeViewModel, което осигурява compile-time проверка на данните и по-добра поддръжка от IDE.

Разделяне на контролери: Действията за дарения бяха извлечени от FundsController в отделен DonationsController, следвайки принципа на единичната отговорност (Single Responsibility Principle).

Екстракция на CSS: Вградените стилове от \_Layout.cshtml бяха преместени в отделен CSS файл (wwwroot/css/kryskatafund.css), подобрявайки разделянето на отговорностите и кеширането на стиловете от браузъра.

Подобрено логиране: Към контролерите и сервизите бяха добавени ILogger инжекции за записване на грешки при обработка на дарения и изпращане на имейли, улеснявайки диагностиката в продукционна среда.

Подобрена обработка на грешки: Реализиран е специализиран ErrorController с отделни страници за различните HTTP грешки (404 Not Found, 403 Forbidden, 500 Internal Server Error), което подобрява потребителското изживяване при проблеми.

## Глава 4 Ръководство за потребителя

В настоящата глава е представено ръководство за основните потребителски сценарии в платформата KryskataFund. Описани са стъпките за регистрация, създаване на кампания, извършване на дарение, както и административните функции.

### 4.1. Регистрация и вписване

За да използва платформата, потребителят трябва първо да създаде акаунт. Процесът на регистрация включва следните стъпки:

1\. Потребителят навигира към страницата за регистрация (Sign Up) чрез главното меню.

2\. Попълва формуляр с електронна поща и парола. Паролата трябва да съдържа минимум 8 символа, включващи поне една главна буква, една малка буква и една цифра.

3\. Системата проверява дали електронната поща вече е регистрирана. При дублиране се показва съобщение за грешка.

4\. При успешна регистрация паролата се хешира с BCrypt алгоритъм и потребителят се пренасочва към страницата за вписване.

5\. За вписване потребителят въвежда своите електронна поща и парола. Системата поддържа както BCrypt, така и наследен SHA256 формат \-- при успешно вписване с SHA256, паролата автоматично се мигрира към BCrypt.

След успешно вписване, в сесията се записват идентификаторът на потребителя, неговият имейл и статус на администратор. Сесията е конфигурирана с 30-минутен таймаут, HttpOnly бисквитки и SameSite=Strict политика за допълнителна сигурност.

### 4.2. Създаване на кампания за набиране на средства

Вписаните потребители могат да създават благотворителни кампании (фондове) чрез следните стъпки:

1\. От главното меню се избира \"Create Fund\".

2\. Попълва се формуляр с: заглавие (до 200 символа), описание (минимум 50 символа), категория (от 8 предефинирани: Education, Health, Animals, Creative, Dreams, Just for fun, Technology, Community), целева сума (от 100 до 1 000 000 лв.), продължителност в дни и снимка (чрез URL адрес или качване на файл).

3\. Описанието на кампанията преминава през HtmlSanitizer филтър, който премахва потенциално опасни HTML тагове и атрибути, предотвратявайки XSS атаки.

4\. При създаване, кампанията получава цветен код според категорията (дефиниран в Constants/CategoryColors.cs) и се визуализира на началната страница.

Създателите на кампании имат достъп до допълнителни функции: публикуване на обновления (Fund Updates), добавяне на етапни цели (Milestones), покана на колаборатори и преглед на аналитика за дарения.

### 4.3. Извършване на дарение

Процесът на дарение е интегриран с платежната система Stripe и преминава през следните етапи:

1\. Потребителят избира кампания и натиска бутона \"Donate\".

2\. Въвежда желаната сума за дарение.

3\. Системата създава Stripe Checkout Session с параметрите на дарението.

4\. Потребителят се пренасочва към защитената Stripe страница за плащане, където въвежда данните на банковата си карта.

5\. При успешно плащане, дарението се записва в базата данни, общата набрана сума на кампанията се актуализира и броят на поддръжниците се увеличава.

6\. Потребителят получава имейл потвърждение чрез Resend API, а създателят на кампанията получава уведомление за полученото дарение.

Платформата поддържа и рекурентни (периодични) дарения, които могат да бъдат отменени от страницата \"My Donations\".

### 4.4. Административен панел

Администраторите на платформата имат достъп до специализиран панел (Dashboard) чрез маршрут /Admin/Dashboard. Достъпът е защитен с RequireAdmin филтър, който проверява администраторския статус в сесията.

Административният панел предоставя следната функционалност:

\- Обзор на платформата: общ брой потребители, кампании, дарения и набрани средства.

\- Последна активност: скорошни дарения и новосъздадени кампании.

\- Разпределение по категории: статистика за броя кампании и набрани средства по категория.

\- Управление на потребители: възможност за изтриване на потребителски акаунти (с предпазна проверка срещу самоизтриване).

При опит на неадминистратор да достъпи административния панел, потребителят се пренасочва към страница \"Forbidden\" (403), вместо към началната страница, което е по-информативно за потребителя.

### 4.5. Допълнителни функции

Освен основните описани по-горе функции, платформата предоставя и редица допълнителни възможности:

Съобщения между потребители: Вписаните потребители могат да изпращат лични съобщения помежду си, включително да споделят линкове към кампании. Системата показва брой непрочетени съобщения и групира разговорите по събеседник.

Следване на кампании: Потребителите могат да следват любимите си кампании чрез бутона \"Follow\" и да ги преглеждат от страницата \"Following\".

Класация (Leaderboard): Публична класация показва топ кампании по набрани средства, топ дарители и топ създатели на кампании.

Персонализация на Buddy: Геймификационен елемент, позволяващ на потребителите да персонализират свой виртуален \"buddy\" с различни аксесоари (очила, шапка, маска).

Вграден уиджет (Embed): Създателите на кампании могат да вградят уиджет за своята кампания в външни уебсайтове чрез iframe, като X-Frame-Options заглавието се премахва специално за тази страница.

Аналитика за създатели: Страница с графики за дарения, списък на топ дарителите и възможност за експорт на данни за дарения в CSV формат.

**Приложение В: Конфигурация на PasswordHasher**

Изходен код на централизирания клас за хеширане на пароли:

public static class PasswordHasher

{

public static string HashPassword(string password)

{

return BCrypt.Net.BCrypt.HashPassword(password);

}

public static bool VerifyPassword(string password, string hash)

{

if (hash.StartsWith(\"\$2\"))

{

return BCrypt.Net.BCrypt.Verify(password, hash);

}

return VerifySha256(password, hash);

}

private static bool VerifySha256(string password, string hash)

{

using var sha256 = SHA256.Create();

var bytes = sha256.ComputeHash(Encoding.UTF8.GetBytes(password));

var computed = Convert.ToBase64String(bytes);

return computed == hash;

}

}
